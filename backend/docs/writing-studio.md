# 写作工作台

写作工作台把单篇热点转换为可人工控制的短帖、Thread 或 X 长文。入口位于热点详情页，第一阶段采用四步闭环：

1. 基于热点原文、快速概览和深度分析提出技术、产业、实践者三个角度。
2. 作者选择角度，并可补充核心判断、真实观察和不同意见。
3. 百炼 `qwen3.7-plus-2026-05-26` 按目标格式生成可编辑草稿。
4. 独立审校调用检查事实归属、论证、模板化 AI 腔和发布格式。

写作与资讯分析使用独立配置 `config/writing.json`。角度和审校使用 JSON Schema，正文使用纯文本；所有来源内容都被明确标记为不可信资料，不能覆盖系统提示。短帖不得超过 280 个字符，Thread 必须包含 4–6 条且每条不超过 280 个字符。后端会执行硬校验，并在第一次不合格时要求模型压缩返工。

## 数据边界

每篇文章最多关联一个 `WritingProject`。第一阶段只保存当前角度、作者输入、当前草稿和当前审校结果：

- 重新生成角度会清空旧草稿和审校。
- 重新生成正文会覆盖旧正文和审校。
- 人工保存正文会覆盖当前正文并清空过时审校。
- 删除孤立文章时，写作项目通过外键级联删除。

这样不会因反复生成建立无限版本历史。后续如果增加版本历史，必须同时增加数量或时间保留策略。

## API

- `POST /articles/{article_id}/writing-project`：获取或创建工作台。
- `GET /writing-projects/{project_id}`：读取当前状态。
- `POST /writing-projects/{project_id}/angles`：生成三个角度。
- `POST /writing-projects/{project_id}/draft`：按选定角度和作者输入生成正文。
- `PATCH /writing-projects/{project_id}`：保存人工修改。
- `POST /writing-projects/{project_id}/review`：审校当前正文。

模型密钥继续通过 `DASHSCOPE_API_KEY` 注入，数据库需升级到 Alembic `0009_writing_projects`。

## 后续阶段

- 第二阶段：多热点合并为同一事件、逐条 claim 与来源映射、可复用声音档案。
- 第三阶段：保存有限版本、记录模型原稿到最终稿的修改差异、从人工选择中学习偏好。
- 第四阶段：长文与 Thread 双向改写、发布前检查清单和效果反馈；不自动发布。
